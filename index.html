A Beginner's Guide to GitHub Pages

Everything you need to know to host your first website — for free — straight from a GitHub repository.

Jump to Step-by-Step Guide →

1. What is GitHub Pages?

GitHub Pages is a free static site hosting service provided by GitHub. It takes the files (HTML, CSS, JavaScript, images) stored in a GitHub repository and serves them as a live website at a public URL — no server, no configuration, no cost.

In simple words: whatever you put inside your GitHub repo, GitHub can automatically show it to the world as a website. Every time you git push changes, the site rebuilds and redeploys — usually within a minute.

Your site will be published at one of these URL patterns:

  • Project site: https://<username>.github.io/<repo-name>/
  • User site: https://<username>.github.io/ (repo must be named <username>.github.io)
  • Organization site: https://<org-name>.github.io/

2. Why Use GitHub Pages?

100% Free

No hosting fees, no credit card required. GitHub serves your site from its global CDN.

Fast & Reliable

Backed by GitHub's global infrastructure — pages load quickly from anywhere in the world.

HTTPS by Default

Every site gets a free TLS certificate from Let's Encrypt — automatically renewed.

Git Workflow

Just git push. No FTP, no build servers to manage. Version control is built in.

Custom Domain

Point your own domain (e.g. myportfolio.com) with a simple CNAME file.

Jekyll Support

Built-in support for the Jekyll static site generator — great for blogs and docs.

GitHub Actions

Use Actions to deploy any modern framework: React, Vue, Angular, Next.js (static export), Astro, Hugo, etc.

Version History

Every change is tracked in Git — roll back to any previous version of your site instantly.

3. Static vs. Dynamic Sites — What GitHub Pages Supports

GitHub Pages hosts static websites only. Understanding this is important before you start.

What is a static site?

A static site is made of pre-built files (.html, .css, .js, images) that are the same for every visitor. The server just serves the file — no processing happens on the server side.

What is a dynamic site?

A dynamic site runs code on a server for each request (PHP, Node.js, Python, Ruby, etc.), often talking to a database, and generates a customized response.

Supported by GitHub Pages

  • HTML, CSS, JavaScript
  • Client-side frameworks (React, Vue, Angular, Svelte — as static builds)
  • Static site generators (Jekyll, Hugo, Astro, 11ty, Gatsby, Next.js static export)
  • Client-side API calls to external services (Firebase, Supabase, REST APIs)
  • Markdown-based blogs and documentation

NOT Supported

  • Server-side languages (PHP, Node.js server, Python/Django, Ruby on Rails)
  • Databases (MySQL, PostgreSQL, MongoDB) — no server to run them
  • Server-side rendering that needs a live server (use static export instead)
  • File uploads / server-side form processing (use a third-party service)
  • Sites larger than 1 GB or with heavy traffic (soft limit ~100 GB/month bandwidth)

Tip: You can still have dynamic behavior on GitHub Pages — you just need to do it in the browser with JavaScript, or call an external API/backend service (e.g. Firebase, Supabase, a serverless function on Netlify/Vercel/Cloudflare Workers).

4. What Kinds of Websites Can You Deploy?

Here are the most common types of sites people host on GitHub Pages:

Personal Portfolio

Showcase your work, resume, and projects. Perfect for developers, designers, and writers.

Project Documentation

Docs for open-source libraries, APIs, and tools — often built with Jekyll, Docsify, or MkDocs.

Personal Blog

Markdown-based blogs powered by Jekyll or Hugo — write posts as .md files.

Landing Pages

Product pages, event pages, "coming soon" pages, marketing microsites.

Resume / CV Site

A single-page online resume you can share as a link with employers.

Documentation Sites

Company handbooks, internal wikis (public), technical guides.

Static Web Apps

Calculators, converters, games, tools written in HTML/CSS/JS or React/Vue/Svelte.

Event / Conference Site

Schedule, speakers, ticket links (via external service like Eventbrite).

Wedding / Invitation Site

Beautiful single-page sites with RSVP forms (using Google Forms or Formspree).

Photo Gallery

Portfolio for photographers, artists, or hobbyists.

Open Source Project Site

Home page for your library or framework, linking to GitHub, docs, downloads.

Educational Tutorials

Course content, interactive lessons, code playgrounds.

5. Step-by-Step Guide: Deploy Your First Site

Follow these steps to publish your very first website on GitHub Pages.

Prerequisites

  • A free GitHub account
  • Git installed on your machine (optional if you use the web UI)
  • An index.html file (this is what the site loads by default)

The 8 Steps

  1. Create a new GitHub repository.
    Go to github.com/new, give it a name (e.g. my-first-site), and set it to Public (private repos need a paid plan for Pages).
  2. Add your website files to the repo.
    Create an index.html at the root. Optionally add styles.css, script.js, and images. Example minimal index.html: 
    <!DOCTYPE html>
<html>
  <head><title>My Site</title></head>
  <body><h1>Hello, GitHub Pages!</h1></body>
</html>
  3. Commit and push your files.
    From the command line: 
    git init -b main
git add .
git commit -m "Initial commit"
git remote add origin https://github.com/<you>/my-first-site.git
git push -u origin main
    Or use "Add file → Upload files" in the GitHub web UI.
  4. Open the repository's Settings.
    In your repo on GitHub, click the Settings tab at the top.
  5. Go to the "Pages" section.
    In the left sidebar under Code and automation, click Pages.
  6. Choose your source.
    Under Build and deployment → Source, select "Deploy from a branch". Then choose the main branch and / (root) folder. Click Save.
  7. Wait for the deployment.
    GitHub will build your site — usually 30 seconds to 2 minutes. You can watch progress in the Actions tab.
  8. Visit your live site!
    Your site is now live at https://<your-username>.github.io/<repo-name>/. Any future git push to main will automatically redeploy it.

Optional: Deploy with a GitHub Action

For frameworks like React, Vue, or Astro, you'll want to build the site first. Use the built-in "GitHub Actions" option in the Pages settings and pick a starter workflow, or write your own .github/workflows/deploy.yml.

6. Using a Custom Domain

Want your site at www.mysite.com instead of *.github.io? It's easy:

  1. Buy a domain from any registrar (Namecheap, GoDaddy, Google Domains, Cloudflare, etc.).
  2. In your repo, add a file named CNAME at the root containing just your domain: www.mysite.com.
  3. At your DNS provider, add these records:
    • CNAME record: www → <username>.github.io
    • Four A records for the apex domain pointing to:
      185.199.108.153, 185.199.109.153, 185.199.110.153, 185.199.111.153
  4. Back in Settings → Pages, enter your custom domain and check Enforce HTTPS.
  5. Wait for DNS to propagate (a few minutes to a few hours) — done!

7. Limits & Things to Know

  • Repo size: should stay under 1 GB.
  • Published site size: soft limit of 1 GB.
  • Bandwidth: soft limit of 100 GB per month.
  • Builds: soft limit of 10 builds per hour (per repo).
  • Private repos: require GitHub Pro / Team / Enterprise for Pages.
  • Prohibited: commercial gambling, illegal content, spam — see GitHub's official docs for full terms.
  • No server code: only static files are served.

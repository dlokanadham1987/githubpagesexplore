A Beginner's Guide to GitHub Pages

Everything you need to know to host your first website — for free — straight from a GitHub repository.

Jump to Step-by-Step Guide →

1. What is GitHub Pages?

GitHub Pages is a free static site hosting service provided by GitHub. It takes the files (HTML, CSS, JavaScript, images) stored in a GitHub repository and serves them as a live website at a public URL — no server, no configuration, no cost.

In simple words: whatever you put inside your GitHub repo, GitHub can automatically show it to the world as a website. Every time you git push changes, the site rebuilds and redeploys — usually within a minute.

Your site will be published at one of these URL patterns:

  • Project site: https://<username>.github.io/<repo-name>/
  • User site: https://<username>.github.io/ (repo must be named <username>.github.io)
  • Organization site: https://<org-name>.github.io/

This site itself is hosted on GitHub Pages at https://dlokanadham1987.github.io/githubpagesexplore/

2. Why Use GitHub Pages?

100% Free

No hosting fees, no credit card required. GitHub serves your site from its global CDN.

Fast & Reliable

Backed by GitHub's global infrastructure — pages load quickly from anywhere in the world.

HTTPS by Default

Every site gets a free TLS certificate from Let's Encrypt — automatically renewed.

Git Workflow

Just git push. No FTP, no build servers to manage. Version control is built in.

Custom Domain

Point your own domain (e.g. myportfolio.com) with a simple CNAME file.

Jekyll Support

Built-in support for the Jekyll static site generator — great for blogs and docs.

GitHub Actions

Use Actions to deploy any modern framework: React, Vue, Angular, Next.js (static export), Astro, Hugo, etc.

Version History

Every change is tracked in Git — roll back to any previous version of your site instantly.

3. Static vs. Dynamic Sites — What GitHub Pages Supports

GitHub Pages hosts static websites only. Understanding this is important before you start.

What is a static site?

A static site is made of pre-built files (.html, .css, .js, images) that are the same for every visitor. The server just serves the file — no processing happens on the server side.

What is a dynamic site?

A dynamic site runs code on a server for each request (PHP, Node.js, Python, Ruby, etc.), often talking to a database, and generates a customized response.

Supported by GitHub Pages

  • HTML, CSS, JavaScript
  • Client-side frameworks (React, Vue, Angular, Svelte — as static builds)
  • Static site generators (Jekyll, Hugo, Astro, 11ty, Gatsby, Next.js static export)
  • Client-side API calls to external services (Firebase, Supabase, REST APIs)
  • Markdown-based blogs and documentation

NOT Supported

  • Server-side languages (PHP, Node.js server, Python/Django, Ruby on Rails)
  • Databases (MySQL, PostgreSQL, MongoDB) — no server to run them
  • Server-side rendering that needs a live server (use static export instead)
  • File uploads / server-side form processing (use a third-party service)
  • Sites larger than 1 GB or with heavy traffic (soft limit ~100 GB/month bandwidth)

Tip: You can still have dynamic behavior on GitHub Pages — you just need to do it in the browser with JavaScript, or call an external API/backend service (e.g. Firebase, Supabase, a serverless function on Netlify/Vercel/Cloudflare Workers).

5. Step-by-Step Guide: Deploy Your First Site

Follow these steps to publish your very first website on GitHub Pages.

Prerequisites

  • A free GitHub account
  • Git installed on your machine (optional if you use the web UI)
  • An index.html file (this is what the site loads by default)

The 8 Steps

  1. Create a new GitHub repository.
    Go to github.com/new, give it a name (e.g. my-first-site), and set it to Public (private repos need a paid plan for Pages).
  2. Add your website files to the repo.
    Create an index.html at the root. Optionally add styles.css, script.js, and images. Example minimal index.html: 
    <!DOCTYPE html>
<html>
  <head><title>My Site</title></head>
  <body><h1>Hello, GitHub Pages!</h1></body>
</html>
  3. Commit and push your files.
    From the command line: 
    git init -b main
git add .
git commit -m "Initial commit"
git remote add origin https://github.com/<you>/my-first-site.git
git push -u origin main
    Or use "Add file → Upload files" in the GitHub web UI.
  4. Open the repository's Settings.
    In your repo on GitHub, click the Settings tab at the top.
  5. Go to the "Pages" section.
    In the left sidebar under Code and automation, click Pages.
  6. Choose your source.
    Under Build and deployment → Source, select "Deploy from a branch". Then choose the main branch and / (root) folder. Click Save.
  7. Wait for the deployment.
    GitHub will build your site — usually 30 seconds to 2 minutes. You can watch progress in the Actions tab.
  8. Visit your live site!
    Your site is now live at https://<your-username>.github.io/<repo-name>/. Any future git push to main will automatically redeploy it.

Optional: Deploy with a GitHub Action

For frameworks like React, Vue, or Astro, you'll want to build the site first. Use the built-in "GitHub Actions" option in the Pages settings and pick a starter workflow, or write your own .github/workflows/deploy.yml.

6. Using a Custom Domain

Want your site at www.mysite.com instead of *.github.io? It's easy:

  1. Buy a domain from any registrar (Namecheap, GoDaddy, Google Domains, Cloudflare, etc.).
  2. In your repo, add a file named CNAME at the root containing just your domain: www.mysite.com.
  3. At your DNS provider, add these records:
    • CNAME record: www → <username>.github.io
    • Four A records for the apex domain pointing to:
      185.199.108.153, 185.199.109.153, 185.199.110.153, 185.199.111.153
  4. Back in Settings → Pages, enter your custom domain and check Enforce HTTPS.
  5. Wait for DNS to propagate (a few minutes to a few hours) — done!

7. Limits & Things to Know

  • Repo size: should stay under 1 GB.
  • Published site size: soft limit of 1 GB.
  • Bandwidth: soft limit of 100 GB per month.
  • Builds: soft limit of 10 builds per hour (per repo).
  • Private repos: require GitHub Pro / Team / Enterprise for Pages.
  • Prohibited: commercial gambling, illegal content, spam — see GitHub's official docs for full terms.
  • No server code: only static files are served.

8. Can I Deploy .NET, React, or Python Apps?

Short answer: Yes — but only the parts that can run without a server. GitHub Pages only serves static files, so anything that needs a running backend (SQL Server, Node.js server, Flask, Django, ASP.NET Core server, etc.) will not work directly. Here's what works for each stack:

React (and Vue, Angular, Svelte)

Fully supported. React apps built with Vite, Create React App, or Next.js (static export) produce a folder of static HTML/CSS/JS — exactly what GitHub Pages needs.

  • Vite + React: npm run build → deploy the dist/ folder.
  • Create React App: npm run build → deploy the build/ folder.
  • Next.js: use output: "export" in next.config.js, then npm run build → deploy the out/ folder.
  • Set base / homepage to your repo path (e.g. "/my-repo/") so assets load correctly.
  • Use React Router in hash mode (#/about) or add a 404.html redirect trick for clean URLs.

Typical GitHub Actions workflow for a Vite/React app:

name: Deploy React to Pages
on:
  push: { branches: [main] }
permissions:
  contents: read
  pages: write
  id-token: write
jobs:
  build-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-pages-artifact@v3
        with: { path: dist }
      - uses: actions/deploy-pages@v4

.NET (Blazor WebAssembly)

Blazor WebAssembly = Yes. Blazor WASM compiles your C# code into WebAssembly that runs in the browser. The published output is 100% static files — perfect for GitHub Pages.

  • Blazor WebAssembly: dotnet publish -c Release → deploy contents of bin/Release/net8.0/publish/wwwroot/.
  • Set the app base href to <base href="/<repo-name>/" /> in index.html.
  • Add a .nojekyll file so GitHub Pages doesn't strip files starting with _ (Blazor uses _framework/).
  • Add a 404.html that redirects to index.html for client-side routing.

.NET on GitHub Pages — Works

  • Blazor WebAssembly (client-side)
  • Static HTML generated by any .NET tool
  • Statiq (a .NET static site generator)
  • DocFX generated documentation

.NET — Does NOT Work

  • ASP.NET Core MVC / Razor Pages (needs a server)
  • Blazor Server (needs a live SignalR connection)
  • Web API projects
  • Any app using SQL Server, EF Core with a DB, etc.

For full .NET apps that need a backend, use Azure App Service, Azure Static Web Apps (great free tier + API support), or containers on Azure Container Apps. GitHub Pages is only for the static UI portion.

Python

Python code doesn't run on GitHub Pages servers — there's no Python interpreter running. But there are two great ways to still use Python:

  • Static site generators: generate HTML with Python on your machine, then publish the output.
    • MkDocs — beautiful documentation sites (very popular).
    • Sphinx — the standard for Python library docs (e.g. NumPy, Django docs).
    • Pelican — Python-based blog generator.
    • Nikola, Lektor — other Python SSGs.
  • Run Python in the browser with Pyodide or PyScript — actual Python code executes client-side via WebAssembly.
  • Jupyter notebooks: convert to HTML with nbconvert or publish with Jupyter Book → static output goes on Pages.

Typical GitHub Actions workflow for MkDocs:

name: Deploy MkDocs to Pages
on:
  push: { branches: [main] }
permissions:
  contents: read
  pages: write
  id-token: write
jobs:
  build-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install mkdocs mkdocs-material
      - run: mkdocs build
      - uses: actions/upload-pages-artifact@v3
        with: { path: site }
      - uses: actions/deploy-pages@v4

Python on GitHub Pages — Works

  • MkDocs, Sphinx, Pelican output
  • Jupyter Book / nbconvert HTML
  • PyScript / Pyodide (browser Python)
  • Any script that generates static HTML

Python — Does NOT Work

  • Flask, Django, FastAPI apps (need a server)
  • Server-side database code
  • Long-running Python processes
  • WebSockets from a Python backend

For live Python apps (Flask/Django/FastAPI), deploy to Render, Railway, Fly.io, Vercel (serverless), Google Cloud Run, or AWS Lambda.

Other Popular Stacks

Node.js / Express

No for server code. Yes for static frontends built with Node tools (webpack, Vite, Parcel).

PHP / WordPress

No. No PHP runtime. But you can use headless CMS + a static generator, or export a WP site to static HTML.

Ruby / Rails

Rails: No. But Jekyll (Ruby) is natively supported — GitHub builds Jekyll sites for you.

Go / Rust

Server: No. But static generators written in Go (Hugo) or Rust (Zola) are perfect — build locally / in Actions, deploy the output.

Java / Spring

Spring Boot: No. But JavaDoc-generated HTML docs or static frontends work fine.

Unity / WebGL Games

Yes! Export your Unity game as WebGL — it produces static files that run great on GitHub Pages.

The Golden Rule

If your app can be built into a folder of static files (HTML/CSS/JS/WASM), GitHub Pages can host it. If it needs a running server process, database, or server-side language interpreter, you need a different host (or split it: static frontend on Pages, API backend on a service like Azure, Render, Vercel Functions, etc.).

9. Frequently Asked Questions

Q: Is GitHub Pages really free forever?

Yes, for public repositories. Private repositories need a paid GitHub plan (Pro, Team, or Enterprise) to use Pages.

Q: How fast does my site update after I push?

Usually within 30 seconds to 2 minutes. You can watch the deployment in the Actions tab of your repo.

Q: Can I use a database?

Not directly. But you can call an external API from JavaScript — services like Firebase, Supabase, Airtable, Fauna, or a serverless function on Vercel/Netlify/Cloudflare Workers work great.

Q: Can I collect form submissions?

Not on Pages itself. Use free services like Formspree, Getform, Netlify Forms (works with a Pages site too), or Google Forms embedded via iframe.

Q: Can I add authentication / login?

Yes — via client-side auth SDKs like Firebase Auth, Auth0, Clerk, or Supabase Auth. All the auth logic runs in the browser.

Q: Can I have multiple sites?

Yes. One user site per account (<username>.github.io) plus unlimited project sites — one per repository.

Q: How do I add analytics?

Add a small JavaScript snippet to index.html. Popular options: Google Analytics, Plausible, Umami, Simple Analytics, GoatCounter.

Q: What happens if my site is popular?

GitHub Pages has a soft limit of ~100 GB/month bandwidth. For most personal sites this is more than enough. Very popular sites should move to a CDN or paid host.

Q: Can I preview locally before pushing?

Yes. Just open index.html in your browser, or run a small local server: python -m http.server or npx serve.
